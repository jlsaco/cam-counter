### 1.1 Modelo de despliegue de infraestructura (convención transversal que este PR DOCUMENTA, no ejecuta)

Hay DOS actores distintos en lo que respecta a Terraform; documéntalos sin ambigüedad en `CLAUDE.md` (y resúmelos en `.claude/README.md` y `CONTRIBUTING.md`):

- **RUNNER MAD (aplica de forma AUTÓNOMA)**: el runner que ejecuta los PRs de infra tiene credenciales AWS amplias en su ENTORNO (jamás commiteadas) y ES QUIEN EJECUTA `terraform apply -auto-approve` de forma autónoma DENTRO de los PRs de infra (PR02 state backend, PR03 OIDC+roles, PR04 media bucket + tablas events/devices + IAM per-Pi, … , PR11 releases bucket), contra la cuenta REAL `950639281773` / `us-east-1`. El apply autónomo del runner NO es humano ni está merge-gated. Cada apply de infra debe ser IDEMPOTENTE (un segundo apply = "0 to add, 0 to change, 0 to destroy") y su Definition of Done exige verificar que los recursos existen DE VERDAD (`aws s3api head-bucket`, `aws dynamodb describe-table`, `aws iam get-role`/`get-open-id-connect-provider`, etc.). Hay UN solo state de producción compartido (`terraform/environments/prod`) con lock en DynamoDB.
- **GITHUB ACTIONS CI (sólo PLAN, nunca apply de infra)**: el CI de GitHub Actions permanece SOLO-PLAN (`terraform fmt`/`validate`/`plan` de sólo lectura vía OIDC). NUNCA ejecuta `terraform apply` — esto evita escalada de privilegios desde PRs no confiables.

IMPORTANTE: esta convención REEMPLAZA cualquier idea previa de que "el apply es responsabilidad humana / merge-gated". El apply de infra lo realiza el runner MAD de forma autónoma; CI sigue plan-only. PR00 en sí mismo NO aplica nada (no hay recursos en este PR); sólo deja escrita la convención para los PRs de infra posteriores.
