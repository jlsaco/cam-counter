## 1) Contexto

`cam-counter` está evolucionando de un detector de personas mono-propósito hacia un PRODUCTO de CONTEO de personas que cruzan una línea-umbral configurable, edge-first (cuenta y persiste en el Raspberry Pi aunque no haya internet), multi-cámara/multi-sitio, con UI local en la LAN y actualización OTA de la flota. Todo vive en este único repositorio (monorepo).

Hechos verificados del borde actual (puedes leerlos):
- El borde funciona en Raspberry Pi 5 + acelerador **Hailo-8** + modelo **YOLOv8s** (`.hef`), con cámara EZVIZ/Hikvision por RTSP.
- El pipeline actual es `detection/yolo_personas_mt.py`: 3 hilos (captura/decode → inferencia Hailo + anotado → encode JPEG/servidor MJPEG en :8080). El wrapper Hailo que debes extraer está en las **líneas 50-68** de ese archivo (función `infer_loop`): crea `HEF`, abre `VDevice`, configura `ConfigureParams`/`InferVStreams`, infiere y lee la salida NMS-por-clase.
- HALLAZGO CONFIRMADO sobre el formato de salida del modelo (documentado en `docs/HALLAZGOS.md`): la salida es **NMS en chip** (`HAILO NMS BY CLASS`, 80 clases COCO, **clase 0 = persona**); cada caja viene como `[ymin, xmin, ymax, xmax, score]` **normalizada 0..1**. En `yolo_personas_mt.py` se desempaqueta así: `ymin, xmin, ymax, xmax, sc = d[:5]`.

Estado de PRs previos en la pila (este PR05 se construye encima de ellos; debes asumir que YA EXISTEN al partir de tu rama base):
- **PR01** movió el sistema de borde a `v1/` (con `git mv`, preservando historia) y creó el esqueleto del repo, el directorio **`contracts/`** con los JSON Schema de los contratos compartidos, y `scripts/version.py`. El antiguo `detection/yolo_personas_mt.py` vive ahora bajo `v1/` (probablemente `v1/detection/yolo_personas_mt.py` — VERIFÍCALO con `git ls-files` / `find` antes de leerlo; no asumas la ruta exacta).
- **PR02/PR03/PR04** crearon infraestructura Terraform (backend de estado, OIDC, bucket de media, tablas DynamoDB de events y device-registry, IAM por-Pi). No tienes que tocar Terraform en este PR.

**Contratos compartidos relevantes para este PR** (definidos como JSON Schema en `contracts/` por PR01; LÉELOS antes de escribir `types.py` para alinear nombres de campo exactos). Los nombres canónicos, en `snake_case`, son:
- **CrossingEvent** (`schema_version=1`): `event_id` (string, hex sha1 determinista en minúsculas de `site_id|device_id|camera_id|track_id|crossing_seq`), `site_id`, `device_id`, `camera_id`, `track_id` (string), `direction` (`'in'`|`'out'`), `label`, `positive_label`, `negative_label`, `line_version` (int), `ts_event_ms` (int epoch ms UTC), `ts_event_iso` (ISO-8601 UTC), `confidence` (float), `clip_key` (string|null), `clip_status` (`'pending'`|`'uploading'`|`'uploaded'`|`'failed'`), `schema_version` (int), `synced` (0/1, solo local), `created_at`.
- **LineConfig**: configuración de la línea por cámara, con endpoints normalizados 0..1, `positive_side` (+1/-1), `positive_label`/`negative_label`, `config_version`/`line_version` (int monótono), `updated_at`.

**Convenciones transversales que DEBES respetar:**
- **Identificadores**: `site_id`, `device_id`, `camera_id` son slugs ASCII en minúsculas que cumplen el regex `^[a-z0-9][a-z0-9-]{1,62}$` (NO `#`, NO `/`, NO mayúsculas, NO vacío, longitud máxima 63). `#` delimita claves compuestas de DynamoDB y `/` delimita rutas de S3, por eso están prohibidos. `camera_id` es globalmente único con la forma `'{device_id}-cam{N}'`. La validación del regex se aplica ANTES de construir cualquier clave. **NUNCA reconstruyas `device_id` a partir de un split de `camera_id`**: `device_id` y `camera_id` se almacenan y pasan como CAMPOS SEPARADOS explícitos.
- **Coordenadas**: TODA la geometría (salida NMS de Hailo, endpoints de línea, overlay, anotación de clip) usa floats normalizados 0..1 relativos al frame ORIGINAL de inferencia, origen arriba-izquierda. Los píxeles nunca aparecen en ningún contrato almacenado. **El Detector reordena la salida de Hailo `[ymin, xmin, ymax, xmax]` al orden de todo el sistema `[xmin, ymin, xmax, ymax]`.**
- **Cero secretos en git**: prohibido cualquier secreto, credencial, password, token o clave de larga vida en el repositorio. No introduzcas IPs, URLs RTSP con credenciales, ni el password de la cámara. `gitleaks` corre en CI; cualquier secreto rompe el build.
- **AWS**: cuenta `950639281773`, región `us-east-1`, prefijo de nombres `cam-counter-`. (No relevante para el código de este PR, pero no introduzcas recursos ni credenciales.)

## 2) Objetivo

Crear `v1/edge/` como un **paquete Python instalable llamado `cam_counter_edge`** (NO un top-level genérico `counter`, para evitar colisión de namespace en el entorno de CI compartido con `v1/api` y `ota`). El paquete debe:

1. Extraer el wrapper de Hailo (HEF/VDevice/InferVStreams) desde el `infer_loop` actual a un **`Detector`** testeable, con **import perezoso (lazy) de `hailo_platform`** para que el módulo se importe en x86 SIN Hailo instalado.
2. Definir tipos compartidos (`Detection`, `Track`) y re-exportar/alinear `CrossingEvent`/`LineConfig` con `contracts/`.
3. Añadir validación de slugs en `identifiers.py`, usada antes de construir cualquier clave.
4. Proveer un **`DummyDetector`** que reproduce una secuencia determinista de detecciones (para testear toda la lógica de conteo en CI x86 sin hardware).
5. Añadir `pyproject`/requirements y un **job de CI (ruff + pytest)** que corre en x86 sin Hailo.

Este PR es **solo scaffold + Detector + DummyDetector + tipos + identifiers + tests + CI**. NO implementa tracking, conteo, SQLite, clips, API ni OTA (esos son PRs posteriores).
