## Arquitectura
cam-counter evolves from a single-purpose Hailo person-detector into an edge-first, multi-camera/multi-site PEOPLE-COUNTING PRODUCT with a LAN UI and an OTA-updatable fleet, all in this monorepo. Five subsystems integrate into one coherent system:

(1) EDGE-COUNTING (Python pkg v1/edge/counter/): the monolithic detection/yolo_personas_mt.py is decomposed into testable stages capture->detect(Hailo)->track(centroid+IoU)->count(half-plane line crossing + hysteresis + per-track idempotency)->present(MJPEG)+clip(circular JPEG buffer -> async MP4)+sink(SQLite). A shared Hailo VDevice serves N cameras under one supervisor process. The Detector is the only Hailo-bound piece; a DummyDetector lets ALL counting logic be tested on x86 CI without hardware. Coordinates are NORMALIZED 0..1 to the original frame everywhere (matching Hailo NMS output, which HALLAZGOS confirms is [ymin,xmin,ymax,xmax,score] normalized; the Detector reorders to [xmin,ymin,xmax,ymax]).

(2) LOCAL-API-UI (v1/api FastAPI + v1/ui React/TS/Vite/Tailwind): ONE Uvicorn process on the Pi serves the JSON/WS API under /api/* and the prebuilt SPA on every other path (same-origin, no CORS, no cloud auth — LAN trust with an optional shared-token gate on writes). MJPEG stays the live-video primitive (per camera) and the configurable counting LINE is a client-side SVG overlay (normalized coords) so the editor drags at 60fps with no round-trip. Config + events + counters live in ONE SQLite DB (WAL) shared with the counting engine via a thin repository layer; config edits hot-reload the line via a monotonic config_version with NO process restart. A FAKE source (env-gated) replays a looped MJPEG + scripted crossings for deterministic Playwright E2E and local dev without a Pi.

(3) CLOUD-AWS-TERRAFORM (terraform/ modules + environments/prod, remote state in S3 + DynamoDB lock): a NEW private TLS-only SSE media bucket for clips (NOT the existing artifacts bucket), a DynamoDB EVENTS table (per-camera PK + time-ordered SK, SITE GSI), a DEVICE REGISTRY table (per-device PK + CHANNEL GSI for canary/stable enumeration), least-privilege per-Pi IAM scoped to each device's own media prefix + DynamoDB key partition, and a GitHub OIDC deploy role. Everything keyed by site_id/device_id/camera_id (lowercase slugs; '#'/'/' forbidden because they delimit DynamoDB/S3 keys).

(4) FLEET-OTA-UPDATE (ota/): PULL-based OTA (no Greengrass/SSM). A versioned, Ed25519/minisign-SIGNED tarball is published to a releases bucket with a per-channel desired-version MANIFEST. A small on-Pi update-agent (systemd timer, Persistent=true) reconciles desired vs current, verifies sha256 THEN signature against a pinned pubkey, installs atomically (releases/<version>/ + 'current' symlink swap), restarts the service, health-checks (service active AND MJPEG :8080/health 200), and auto-rolls-back to last_good on failure. Mutable state (SQLite DB, line config, native blobs) lives in shared/ and survives every upgrade/rollback. Offline-tolerant: a disconnected Pi keeps running last-good and converges to the channel's CURRENT version on reconnect (one jump, never a queue).

(5) CICD-VERSIONING (.github/workflows, scripts/): annotated git SemVer tags (vX.Y.Z, -rc.N) are the SINGLE source of truth via scripts/version.py (git describe). Path-filtered PR CI (ruff/pytest, eslint/tsc/vitest/vite build, terraform fmt/validate, gradle build, gitleaks, edge-bundle dry-run). On tag: build+sign+publish to releases bucket and auto-point the canary channel manifest. A separate Environment-gated promote/rollback workflow only REPOINTS the channel manifest to an already-published, canary-soaked version (no rebuild) — symmetric, fast canary->fleet promotion and rollback.

## Convenciones
- Identifiers: site_id, device_id, camera_id are lowercase ASCII slugs matching ^[a-z0-9][a-z0-9-]{1,62}$ (NO '#', NO '/', since '#' delimits DynamoDB composite keys and '/' delimits S3 paths). camera_id is globally unique = '{device_id}-cam{N}'. event_id is deterministic lowercased-hex sha1(site_id|device_id|camera_id|track_id|crossing_seq). Validate the regex at device registration AND in the edge agent before constructing any key.
- Coordinates: ALL geometry (Hailo NMS output, line endpoints, overlay, clip annotation) uses normalized 0..1 floats relative to the ORIGINAL inference frame, origin top-left. Pixels never appear in any stored contract. The Detector reorders Hailo's [ymin,xmin,ymax,xmax] into the system-wide [xmin,ymin,xmax,ymax].
- Line-crossing direction sign: side(P) = sign(cross(B-A, P-A)); a crossing = side flip across consecutive centroids; the per-camera positive_side flag (+1/-1) selects which flip counts as 'in'. The wire/stored direction value is exactly 'in'|'out'; 'subieron'/'bajaron' are positive_label/negative_label, and UI positive/negative / cloud up/down are display mappings only. A unit test feeds a known crossing and asserts the sign — this is the single most fragile shared contract.
- CrossingEvent + LineConfig + Counters + DeviceInfo + Camera + WsEnvelope are defined ONCE as Pydantic models in v1/api/schemas.py and mirrored as JSON Schema files under contracts/. Any field rename is a BREAKING change requiring schema_version bump and a coordinated PR; CI snapshots /api/openapi.json so drift fails the build.
- Versioning: annotated git SemVer tags (vX.Y.Z, prereleases -rc.N) are the ONLY source of truth; scripts/version.py derives it via git describe (no committed VERSION file). release/version CI jobs MUST use actions/checkout fetch-depth:0 or git describe yields wrong versions. The same string flows through bundle-manifest, channel manifest, device-registry, and /api/device app_version.
- Buckets: THREE distinct S3 buckets, never conflated — (1) cam-counter-rpi-artifacts-950639281773 = EXISTING ops-binaries backup only, RESERVED/do-not-touch; (2) cam-counter-media-950639281773 = NEW product media (clips/gifs/snapshots), key media/{site_id}/{device_id}/{camera_id}/{yyyy}/{mm}/{dd}/{event_id}.{ext}; (3) cam-counter-fleet-releases-950639281773 = NEW OTA artifacts + channel manifests. All NEW buckets: private, BlockPublicAccess all-true, SSE-S3, bucket-owner-enforced, TLS-only deny policy.
- Zero secrets in git: NO long-lived AWS keys; GitHub Actions assume AWS via OIDC (provider token.actions.githubusercontent.com, trust scoped to repo:jlsaco/cam-counter with ref/tag/environment sub conditions). Per-Pi creds are short-lived STS from a per-device least-privilege role. Camera credentials (currently hardcoded RWCHBY in start_detection.sh and README) are templated out of any published artifact and sourced from env/SSM/SQLite; gitleaks runs in CI. The artifact builder strips device-specific config and absolute /home/pi paths.
- Terraform: modules under terraform/modules/<service>, single live root terraform/environments/prod, remote state S3 + DynamoDB lock (PK exactly 'LockID'), provider pinned (~> 5.x) with default_tags {Project=cam-counter, ManagedBy=terraform, Env=prod}, prefix cam-counter-, account 950639281773, us-east-1. Directory-per-env (not workspaces). DynamoDB tables PAY_PER_REQUEST with PITR. apply is human/merge-gated; CI only does fmt/validate and OIDC read-only plan.
- On-device layout: immutable /opt/cam-counter/releases/<version>/, atomic activation via /opt/cam-counter/current symlink, mutable upgrade-surviving state in /opt/cam-counter/shared/ (SQLite DB, line config, CAM_IP, box64 sysroot, SDK lib). Heavy host deps (box64, ffmpeg, hailo-all, the .hef) stay OUTSIDE the OTA payload, installed at provisioning. Atomic installs: extract to temp then rename; never delete last_good until the new release is health-confirmed; .part files discarded on boot.
- Edge SQLite is the single persistence contract: WAL mode, busy_timeout, short transactions, single writer connection, migrations via PRAGMA user_version. Tables events (UNIQUE event_id, synced flag), counters (PK camera_id/day_utc/direction), clip_uploads (upload queue), camera_config (config_version, updated_at). cloud-sync reads events WHERE synced=0, uploads clip + conditional-puts the event, sets synced=1.
