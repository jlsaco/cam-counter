# ROADMAP — cam-counter AWS IoT Core

Roadmap CORREGIDO para migrar cam-counter de su sync directo (DynamoDB/S3 via SigV4) hacia identidad por certificado X.509 + AWS IoT Core, mas una consola de flota Next.js/Amplify. Se entrega como 18 PRs APILADOS (cada uno ramifica sobre la rama del WP anterior; el primero sobre main; merge nunca squash). HECHO CLAVE de despliegue: MAD aplica 'terraform apply' de forma AUTONOMA usando la identidad admin 'raspberry' (el IAM user con AdministratorAccess cuyo access key estatico es el ~/.aws/[default] de la Raspberry Pi, que 'mad serve' hereda al no definir variables AWS_*). Por tanto TODOS los WPs de infra aplican bien bajo MAD: el 'gap de permisos del rol cam-counter-gha-deploy' es NO BLOQUEANTE (MAD no usa ese rol; ese rol es solo para GitHub Actions plan-only) y se deja como nota de hardening futuro, no como WP bloqueante. GUARDARRAIL CRITICO: NINGUN WP borra ni desactiva el user 'raspberry', su access key, ni el ~/.aws/credentials de la Pi (eso romperia MAD). El WP de 'corte' se RE-SCOPEA: hace que el PROCESO EDGE deje de usar credenciales AWS directas (el edge pasa a publicar por MQTT con su certificado y sube clips via IoT Credential Provider / role alias, sin llaves estaticas), dejando intacta la identidad de despliegue de MAD. El prefijo de entorno canonico es CAMCOUNTER_* (el que ya usa el codigo); el feature flag de transporte es CAMCOUNTER_SYNC_TRANSPORT con valores direct|iot (default direct). Los contratos en contracts/ (crossing_event.schema.json, line_config.schema.json, device_registry_item.schema.json) se respetan VERBATIM: el payload MQTT de eventos ES el CrossingEvent del contrato (el clip va aparte a S3; el evento lleva clip_key) y la Lambda de ingesta valida contra ese schema. La IoT policy de minimo privilegio acota topics por ${iot:Connection.Thing.ThingName} (client-id MQTT == Thing name), NO por lookups de atributos del Thing. Se inserta temprano un spike de PoC Hailo-en-Docker en ARM64 real con criterio GO/NO-GO y plan B nativo; el corte (MQTT) y la dockerizacion quedan DESACOPLADOS (la dockerizacion depende del PoC, no del corte). Orden: fundaciones doc/contratos -> IAM/credential-provider -> ingesta serverless -> iot-core -> provisioning + PoC Hailo -> consola (cognito/api/Next.js/Amplify) -> migracion segura del edge dual-run -> shadow -> corte del camino directo del edge -> dockerizacion -> hardening/observabilidad/runbooks. Todo el Terraform es aditivo/monotono: MAD aplica auto-approve como raspberry-admin y aborta ante destroy/replace; recursos por-flota van en Terraform, recursos por-device (thing, cert, item registry) los crea el script de provisioning fuera del state.

## Fases

- **P0 Fundaciones: nombres y contratos** — Fijar el estandar de nombres canonico (doc) reconciliando divergencias entre specs SIN contradecir el codigo (prefijo de entorno CAMCOUNTER_*), y reconciliar/validar los contratos (crossing_event, line_config) verbatim con un gate de CI, antes de escribir Terraform o tocar el dispositivo. Cero riesgo, solo doc + tests.
- **P1 Plano de control IoT, ingesta serverless y provisioning** — Desplegar el camino MQTT->IoT Rule->Lambda->DynamoDB (iam-lambda, iot-credential-provider, lambdas de ingesta, iot-core con policy acotada por ThingName, provisioning de certs + script, fleet provisioning) y validar el PoC Hailo-en-Docker en ARM64 real (GO/NO-GO + plan B nativo). El dispositivo aun no publica; la infra queda lista y validable de extremo a extremo. MAD aplica como raspberry-admin.
- **P2 Consola de flota (lectura)** — Construir el plano de lectura: Cognito de operadores, API HTTP + lambdas de lectura/presign, app Next.js y hosting Amplify, con reproduccion de clips por presigned URL. Solo lee datos que ya escribe la Lambda de ingesta; no toca el edge.
- **P3 Migracion segura del dispositivo** — Refactor del edge a publicacion MQTT en DUAL-RUN (feature flag CAMCOUNTER_SYNC_TRANSPORT=direct|iot), conectar Device Shadow para line-config/comandos, validar paridad MQTT==directo en la INGESTA, y cortar el camino directo del PROCESO EDGE (pasa a MQTT + IoT Credential Provider, sin credenciales AWS directas). NO se toca la identidad de despliegue raspberry.
- **P4 Dockerizacion y operacion** — Dockerizar el dispositivo (Hailo passthrough), desacoplada del corte y apoyada en el PoC GO/NO-GO con plan B nativo, y cerrar con observabilidad end-to-end (alarmas, DLQ, dashboards, lifecycle events), endurecimiento de la consola/flota y runbooks de provisioning/rotacion/revocacion de certs e incidentes.
